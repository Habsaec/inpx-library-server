# Развёртывание в продакшене

Краткий чеклист безопасной эксплуатации INPX Library Server за обратным прокси или в Docker.

## Обязательно

1. **HTTPS**  
   Выдавайте сайт только по TLS (nginx, Caddy, Traefik, Cloudflare и т.д.). Иначе сессионная cookie и трафик читаются в открытом виде.

2. **Защищённая cookie сессии**  
   После настройки HTTPS включите:
   ```env
   SESSION_SECURE_COOKIE=1
   ```
   Иначе браузер может отправлять cookie и по HTTP.

3. **Секрет сессии**  
   Задайте стойкий неизменный секрет (или полагайтесь на `data/.session-secret` на постоянном томе):
   ```env
   SESSION_SECRET=<openssl rand -hex 32>
   ```
   Смена секрета инвалидирует сессии и ломает расшифровку сохранённых в БД секретов SMTP (их нужно ввести заново в админке).

## За обратным прокси

4. **Trust proxy**  
   Включайте только если запросы к Node действительно идут через доверенный прокси, который **перезаписывает** `X-Forwarded-For` / `X-Real-IP`, а не доверяете сырому заголовку от клиента:
   ```env
   TRUST_PROXY=1
   ```
   При неверной настройке возможны ошибки в ограничении попыток входа (IP) и лишняя доверительность к подделанным заголовкам.

## Опционально

5. **Docker / логи**  
   `EVENTS_STDOUT=true` (по умолчанию) дублирует события в stdout — удобно для `docker logs`.

6. **CSP**  
   В `src/middleware/security-headers.js` политика допускает `unsafe-inline` для скриптов и домены Google (reCAPTCHA при необходимости). При смене виджетов проверьте, что страница не ломается в консоли браузера.

## API для интеграций

JSON-ответы с ошибками содержат стабильное поле **`code`** (например `UNAUTHORIZED`, `BATCH_NO_BOOKS`) и локализованное **`error`**. Клиентам лучше ориентироваться на `code`.

Пакетная выгрузка ZIP выставляет заголовки:
- `X-Batch-Requested` — id в запросе  
- `X-Batch-Included` — файлов добавлено в архив  
- `X-Batch-Skipped` — пропущено (нет книги или ошибка подготовки)

Ответ **`POST /api/send-to-ereader/batch`** при успехе может содержать `requested`, `attached`, `skipped`, `partial`.
